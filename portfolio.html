Why I built it

“보청기·무선 이어폰 사용자는 외국어 강연·대화에서 실시간 통역 수단이 마땅치 않다.”

기존 통역 앱은 발화가 끝나야 번역되는 turn-based라 동시통역이 어렵고, 보청기(MFi) 오디오 라우팅 문제로 실사용이 막힙니다. StarLink는 발화 중에도 번역이 흐르는 연속 스트리밍 통역을 무선 이어폰·보청기로 직접 전달해 이 공백을 메웁니다. 기획자가 직접 문제를 정의하고, AI 모델(Codex·Gemini·Claude)을 활용해 iOS 클라이언트부터 Python 백엔드·인프라까지 1인으로 구현했습니다.

What it does

핵심 기능

핸즈프리로 외국어 대화에 자연스럽게 참여하도록, 통역·자막·기록을 하나의 흐름으로 묶었습니다.

🎧

모든 무선 기기 지원

에어팟, 갤럭시버즈, 일반 블루투스 이어버드, MFi 보청기를 동일하게 처리합니다. 기기별 설정 없이 연결된 무선 기기로 통역 음성이 자동 라우팅됩니다.

💬

실시간 동기 자막

음성 출력과 동시에 번역 자막이 화면에 표시됩니다. 입력 언어를 자동 감지해 [영어]·[일본어] 글래스모피즘 배지로 함께 보여줍니다.

📝

회의록 기록 모드

통역 모드와 별개로, 한국어 포함 모든 언어를 0ms 지연으로 받아쓰는 STT 전용 모드. 대화 히스토리를 .txt로 내보내 공유할 수 있습니다.

🔑

BYOK 설정 UI

톱니바퀴 버튼으로 LiveKit URL·토큰을 직접 입력해 UserDefaults에 저장. 서버 비용 없이 개인 키로 동작하는 앱스토어 배포 시나리오를 지원합니다.

🔒

토큰 서버 보안

aiohttp 기반 토큰 서버가 1시간 TTL 단기 JWT를 발급. iOS는 실행 시 동적 토큰을 받아오고, 없으면 안전하게 폴백합니다.

⚡

저지연 WebRTC

LiveKit Cloud의 미디어 라우팅 엔진으로 양방향 오디오·데이터를 스트리밍. 마이크 → 통역 → 이어폰까지 1초 미만으로 끊김 없이 흐릅니다.

Engineering Deep-Dive

해결한 4대 기술 난제

실시간 오디오 + WebRTC + 음성 AI가 충돌하는 지점들을 하나씩 깨면서 만든 프로젝트입니다.

01

블루투스 & MFi 오디오 라우팅 제어 AVAudioSession · WebRTC

⚠ 문제

LiveKit의 자동 오디오 구성이 AVAudioSession 카테고리를 재설정해, 연결된 이어폰 대신 아이폰 내장 리시버로 오디오가 라우팅. 직접 세션을 조작하면 WebRTC 세션 관리와 충돌해 비동기 오디오 스레드 데드락 발생.

✓ 해결

출력 경로 제어를 raw 오버라이드 대신 LiveKit 공식 API(isSpeakerOutputPreferred)로 통제. 보청기 연결 시 스피커 우선을 끄고 짧은 블루투스 정착 지연을 넣어 AUIOClient_StartIO (-3001) 데드락을 회피.

02

음향 피드백 루프 방지 Echo Cancellation

⚠ 문제

이어폰으로 재생되는 통역 음성이 마이크로 다시 새어 들어가, AI가 자기 통역을 듣고 재통역하는 무한 에코 루프(then, and then… 도배) 발생. 마이크 트랙을 물리적으로 뮤트하면 WebRTC 오디오 디바이스 락 위험.

✓ 해결

마이크 트랙을 물리적으로 건드리지 않고, Silero·VAD·turn-taking을 전부 제거. Gemini의 자동 발화 감지를 끄고(AutomaticActivityDetection(disabled=True)) 오디오를 연속 스트림으로 처리하며, generate_reply를 가로채 ActivityEnd 전송을 막아 모델이 자기 출력을 재통역하지 않게 함.

03

음성-자막 모달리티 동기화 Gemini Live

⚠ 문제

음성-대-음성 모델은 기본값에서 전사(transcription) 스트림을 보내지 않아 자막이 번역 생성 중 🔄 상태에 멈춤. SDK가 같은 비동기 제네레이터를 이중 소비하며 스트림이 프리징되는 문제도 동반.

✓ 해결

세션 초기화 시 입·출력에 AudioTranscriptionConfig()를 주입해 전사 토큰을 강제 방출. emit 몽키패치와 스트림 래퍼로 부작용 없이 자막 텍스트만 복제 전송해 오디오·자막을 동시 복구.

04

무지연 입력 언어 감지 0ms Unicode Parser

⚠ 문제

다국어 대화의 원문 언어를 표시하려면 언어 감지가 필요한데, 외부 번역 API를 쓰면 허용 불가한 네트워크 지연이 추가됨.

✓ 해결

유니코드 블록(한글·가나·CJK 한자·라틴)을 분석하는 0ms 경량 파서 detect_language를 백엔드에 구현. 언어 코드를 LiveKit 데이터 채널로 전송해 실시간 언어 배지를 렌더링.

backend/agent.py — 0ms 유니코드 언어 감지 
def detect_language(text: str) -> str:
    if not text:
        return "unknown"
    text = text.strip()

    # Hangul (Korean)
    if any('가' <= c <= '힣' for c in text):
        return "ko"
    # Hiragana / Katakana (Japanese)
    if any('぀' <= c <= 'ヿ' for c in text):
        return "ja"
    # Chinese Hanzi
    if any('一' <= c <= '鿿' for c in text):
        return "zh"
    # English / Latin
    latin = sum(1 for c in text if 'a' <= c.lower() <= 'z')
    return "en" if latin > len(text) * 0.3 else "unknown"

Build Journey

시행착오 포스트모템

실제 개발 로그(progress.md)에 남긴, 막힌 지점과 그걸 뚫은 방법의 기록입니다. 결과보다 과정이 더 많은 걸 말해줍니다.

마이크 에코 루프 — then, and then… 무한 도배

초기엔 VAD로 마이크를 즉시 여닫자 통역 잔향이 재유입돼 번역이 정지. 백엔드 상태 메시지 기준으로 바꿨더니 조기 언뮤트로 또 막힘.

→ 물리적 mute·VAD를 전부 버리고, 자동 발화 감지 비활성화 + generate_reply 차단으로 스트리밍 전용 전환

오디오가 보청기 대신 폰 리시버로 새는 문제

자동 구성·커스텀 콜백 모두 WebRTC가 세션을 수화기 모드로 덮어써 무음 발생. overrideOutputAudioPort 직접 호출은 OSStatus -50 에러.

→ 자동 구성(isAutomaticConfigurationEnabled=true)에 위임하고 isSpeakerOutputPreferred로 출력 경로 제어

비동기 스트림 이중 소비로 자막·오디오 동반 정지

백엔드와 SDK가 같은 message_stream 제네레이터를 동시에 소비하며 태스크 프리징.

→ emit 몽키패치 + intercept 래퍼로 부작용 없이 자막만 복제, 양쪽 완전 복구

인터럽션 차단 시 세션 시작 즉시 크래시

초기엔 서버 사이드 발화 감지 + allow_interruptions=False로 가로채기를 막으려다 SDK 내부 ValueError·오디오 폐기 크래시.

→ 모든 speech handle에 allow_interruptions=True 부여 + Gemini 자동 발화 감지 비활성화(로컬 VAD 없이)로 안정화
